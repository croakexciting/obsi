
## 1 简介

操作系统模块或者操作系统服务提供操作系统中所有核心功能，比如设备驱动，文件系统，内存管理模块，甚至是调度器。

宏内核例如 Linux，这些模块和 kernel 是同一个进程，共享地址空间，栈空间。任意一个模块都可能会造成 kernel panic，使得内核模块开发需要经过长时间大量测试，否则很容易出现严重后果。针对这个问题，主流方向是使用 **安全语言 Rust**，**域隔离**，比如 Readleaf, 陈林峰同学的工作。

微内核如 seL4, QNX, 操作系统服务通常是独立进程，相对来说更加安全。安全性的提升主要基于 安全语言 Rust，更多的工作集中在提升 IPC 性能，突破特权级隔离。

在这些工作基础上，可以进一步提升模块的安全性。每个模块都是由多个 线程/协程 组成的，我们可以将其归纳为执行流。如果我们可以针对每个执行流进行保护，那么自然实现了对模块的保护。

同时，目前工作主要针对的是模块之间的隔离，即强调模块之间互相不干扰，但是并不在意某个模块的稳定性。因此，我们的工作中心在于提高每个模块的稳定性。通过 **主动检测** + **主动恢复** 的方式，在发生 panic 之前，主动重启执行流，提高模块鲁棒性。

### 2 工作目标

- 基于主动检测和恢复，实现用于异步微内核场景的执行流保护功能，提升系统模块稳定性
    
- 结合之前 reL4 和 arceos 的工作，让系统模块实现更简单，能用于多种微内核系统
    
- 应用到 Linux kernel module 中，提高 kernel module 的稳定性
    
- 将 soft-lock 用于异步微内核场景，看看能提供什么额外优点
    

## 3 核心思路

该思路来源于 Hopter 设计，在 rust 已经实现大部分内存安全的基础上，增加了 **主动栈检查** 功能，在每次栈分配之前检查该次分配是否会造成栈溢出，如果溢出，怎会主动触发 panic。

无论是主动触发 panic，还是 rust 原本造成的 panic，都会触发 panic handler，在其中执行 unwinder 函数，进行主动恢复，重启执行流。

在设想中，该方案可以用于微内核和 Linux kernel module 中。

### 3.1 栈检查

目前考虑两种实现模式

- 在编译器中做修改，每个函数之前增加一个 prelude，计算这次执行函数是否会造成栈溢出。该方案已经在 hopter 中经过验证，可行性没有问题。但是问题是修改编译器影响面比较大，不够灵活。同时如果不能合入 rust 主分支，需要针对每个版本做 patch，难以在实际场景使用。
    
- 使用 rust 库的方式，实现只针对某些函数进行栈溢出检查。这样做更加干净，灵活度比较高。问题是如何知道每个函数执行时需要分配多少占空间，需要探索下能否实现。
    

### 3.2 主动恢复

主动恢复基于 rust 的 unwinder 机制实现，在 panic 中进行栈展开，获取导致 panic 的控制流信息，然后重启该控制流。类似功能在多种安全系统设计中都有实现，如 hopter，readleaf 等等，可以参照。

### 3.3 soft-lock

soft-lock 是 hopter 实现的同步互斥机制，其特点就是永远不会卡在加锁的地方，而是将锁之后执行程序 defer 到锁释放后执行，有点像中断的上下部分。

该实现的好处是，因为永远不会卡死，所以加锁的时候不需要禁用中断，所以中断完全没有延时。但是目前还没想清楚怎么结合。

## 4 微内核

微内核系统中，有很多需要开发的操作系统服务。目前的方案主要针对如何突破服务之间的地址空间隔离，使用比 IPC 更轻量的调用方式。

我们可以将重心放在如何增强这些模块的稳定性，使用上述的主动检测和恢复的方法可以实现。尤其是一些操作系统服务是运行在内核态，这种重要的服务更加需要提升可靠性。

计划是先在 seL4/reL4 平台上尝试实现，因为之前已经有了一些工作基础，

除了提升可靠性，还希望可以基于 arceos 更高效可靠的实现操作系统服务，适配一些主流微内核系统。

## 5 Linux

陈林峰同学已经实现了 kernel module 热更新和隔离功能，所以计划是在他的工作上继续，实现对于模块中执行流的保护。

如果经过验证，可以使用库的方式实现主动检测，也会尝试下在 rust for Linux 官方版本上实现该功能。

## 6 工作计划

下列计划按照从前到后顺序实施，计划从主动检测功能开始，验证两种实现方案，并且先应用于微内核场景。

- seL4 上，主动检测功能实现，验证两种方案
    
- seL4 上，主动恢复功能实现
    
- 基于 Arceos，在 seL4 上实现带主动验证和恢复功能的操作系统服务
    
- 在 Linux kernel module 中，实现主动检测和恢复功能
    
- 探索主动检测恢复在异步微内核中的应用
    
- 探索软锁在异步微内核中的应用
    
- 将基于 Arceos 实现的操作系统服务移植到其他微内核系统上
