### 阅读后思考

1. 对于 rust 内存安全的增强修改是否有意义用在更复杂系统上，这种通过检查发现溢出的功能可以直接在现场发现问题，可以大大降低问题排查难度，即使不考虑恢复之类，也很有价值，不知道目前有没有类似工作。同样的，这种方式能否用于其他内存问题场景，rust 对于这些已经支持很完善，那么能否用在 c 上。
2. soft-lock 能否移到更复杂的 OS 中，hopter 中只是用其解决中断和任务之间的同步，任务和任务之前根据其设计是不会发生锁竞争的。用法比较简单，如果移到更复杂 OS 中会有挑战，需要找到 soft-lock 在其他操作系统中的应用场景
3. 目前基于 rust 的各种方法，把操作系统服务做的越来越安全，微内核这种完全隔离的架构是否还有意义，可以考虑将可信，或者经过测试认证的模块放到一起。只是说客户或者开发的驱动服务之类的还是在用户态，弱化内核和服务之间的隔离界限

### 向老师意见

1. 在 C 中使用宏的方式实现插桩，进行栈溢出检查
	1. 个人认为 rust 已经覆盖了大部分内存安全功能，如果用 C 做，虽然可以用各种方式实现如栈溢出检查这样的单个功能，但是实现完整内存安全相当于重新实现一遍语言，不现实
	2. 如果能找到某个特定场景只需要实现单个内存安全功能，在 C 上实现还是有价值，毕竟性能相比 rust 是要高
2. 现在 rust 上重现该工作，并在某个系统，如 QNX，seL4 上运行
3. soft-lock 机制很牛，有点像 RCU 这种类型，想象可以用在什么场景
4. unwind 之后重启任务类似热升级，理论上可以用这种方式主动重启任务
5. 任务故障检查和重启在微内核上有什么用
	1. 比如不同任务之间共享锁，一个任务如果直接崩溃，造成死锁。类似的任务现场处理问题。
