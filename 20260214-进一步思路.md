### 2. Hopter 实现学习

栈检查功能实现在编译器中，需要对 Rust 和 LLVM 进行修改，确认了以下几点
- 在每个函数之前都加了 Prologue，检查这次分配是否会造成栈溢出
- TODO，还要再具体看下实现

目前发现移植会遇到的问题
- 栈检查功能依赖 LLVM 的 segmented-stack flag，LLVM 每次增加这个选项，会自动将该功能带入。但是这个 flag 一般很少用在 Linux, QNX, seL4 等通用操作系统，如果移植的话，需要换种方式实现对编译器的修改。
- hopter 中检测和恢复是针对一个 rtos 任务，也就是一个线程。但是移植到无论是 OS 服务 还是 Linux module，都是针对类似一个进程 （多个线程的组合）。这种时候恢复该如何定义
- 如果在 Linux Module 中的话，需要用修改后的编译器重编整个 kernel，才能保持一致。需要想办法去避免，不然 Linux 中所有函数都加上 prologue 的话，影响范围太大了，感觉不太对。
- 目前的实现是针对 armv6 平台，同样要修改。
- 如上，可以借鉴 hopter 实现，但是基本是要重新自己实现对编译器的修改，没办法直接拿来用

### 3. 微内核

QNX 中允许应用线程运行在内核态 EL1，原理是 aarch64 支持 EL1t 和 EL1h 两种模式，其中 EL1t 使用用户栈，EL1h 使用内核栈。EL1t 状态下的线程和普通线程几乎一样，只是可以执行一些特权级操作。

我感觉这个功能可以和栈检查和自动恢复很好的结合。将一些服务运行在内核态，通过栈检查功能保持其稳定可恢复，整理以下可扩展的点

- QNX 内核态服务 + 栈检查可恢复
	- Rust 版本，实现完整 hopter 内存安全功能
	- C 版本，需要继续探索能实现哪些功能
- QNX 内核态线程仍然使用 svc 进入 kernel，测试下来发现和用户态线程 syscall 时间差不多，没有性能优势。考虑下类似 redleaf 和 陈林峰 等做的工作，将内核态服务调用优化下。感觉可以提升性能，但是具体怎么实现还不知道。
- 考虑到 QNX 是一个闭源系统，不可以公开实现，因此想在 seL4 上做相同实现
	- seL4 没有内核态线程，需要实现
	- 同样优化内核态服务调用，不使用 svc 进入 kernel，或者可以由 kernel 直接调用，类似函数调用。这个是否可以借鉴 廖东海 的工作？
	- Rust 版本，实现完整 hopter 内存安全功能 
- 需要找到一个十分具体的使用场景，一个 benchmark，涉及大量通信或者交互，来验证改动的先进性。

以上是我在微内核中想到能做的工作，这样可以把 主动恢复 + 微内核 IPC 优化 综合起来，认为是主动恢复的一个实际使用场景。
### 4. Linux

向老师之前提到，可以用在 Linux kernel module 中。应用场景是确定了，但是目前我还有以下几个问题没想明白
- 检查什么栈？整体公用的内核栈吗，这似乎超出了一个模块的范围。那是检查这个模块中的内核线程的栈吗？这样是否足够
- 模块主动恢复怎么理解，没想明白，准备看下 陈林峰 的工作，我理解只有一个 panic handler 吧
- 修改 llvm 编译器影响整个内核，怎么处理呢

关于 Linux 中的模块，最近简单了解了下 陈林峰 的工作，发现他没有使用 kernel 自带的 rust 相关功能，而是基于 linux-kernel-module-rust 。这么做的好处是，不需要使用相同的编译器编译整个 kernel，只编译某个 kernel module 即可，似乎和修改 toolchain 的路线更加契合。
