## 第一阶段：先跑通“panic → unwind → catch”的最小闭环（第 3–6 周）

**阶段目标**：先证明 reL4 在普通任务上下文里，能够把 Rust 的 `panic!()` 正确转换成可恢复的 unwind 流程。这个阶段只验证“任务 panic 后系统还能继续工作”，先不引入栈溢出、任务重启、ISR panic 等复杂路径。

**这一阶段做完后，应该能明确回答下面 4 个问题**：
- 任务里调用 `panic!()` 时，是否真的进入了内核的 `panic_handler`
- `panic_handler` 是否能把 panic 转成 `unwinding` 可处理的异常对象
- `catch_unwind` 是否能稳定收到 `Err`，而不是被直接 abort 掉
- panic 过程中栈上的资源是否都能正常执行 `Drop`

### Week 3：先把最小链路跑通

| # | 任务 | 设计文档章节 |
|---|------|------|
| 1.1 | 先用 `aarch64-baremetal-unwind/` 或最小原型确认 `unwinding` crate 在当前工具链和裸机环境下可用 | 无直接对应；这是在正式接入内核前验证依赖可用性的前置任务 |
| 1.2 | 编写最小 `panic_handler`：接收 `&PanicInfo`，打印 panic 位置/消息，然后调用 `unwinding::panic::begin_panic(Box::new(()))` | Section 4.1 |
| 1.3 | 实现统一任务入口 `task_run(task_fn)`：内部包一层 `catch_unwind(task_fn)`，将结果统一分成 `Ok` 和 `Err` 两条分支 | Section 4.1、Section 6 |
| 1.4 | 补齐最小调试日志：在 `panic_handler`、`task_run`、`Err` 分支分别打印固定标记 | 无直接对应；这是为了让后续 QEMU/实机排障可观察 |

### Week 4–5：补上线程创建和端到端验证

| # | 任务 | 设计文档章节 |
|---|------|------|
| 1.5 | 实现内核线程创建原语 `kernel_spawn_thread(task_run, task_fn)`：分配独立栈、初始化 TCB，并把任务入口统一指向 `task_run` | Section 6 |
| 1.6 | 在 QEMU 上做端到端验证：任务 `panic!()` → `panic_handler` → `begin_panic` → unwind Phase 1/Phase 2 → landing pad → `task_run` 的 `Err` 分支 | Section 4.1、Section 4.2、Section 6 |
| 1.7 | 增加 per-task `TASK_UNWINDING` 标志（放在 TCB 里），用于标记当前任务是否正处于展开中 | Section 5.2；用于后续 double panic 保护 |
| 1.8 | 做多任务隔离验证：同时运行两个任务，一个任务 panic，另一个任务继续正常执行 | 无直接对应；这是对任务隔离和调度影响面的补充验证 |

**第一阶段里程碑**：在 QEMU 上，任务 panic 后能自动完成 Drop cleanup，并且 `task_run` 稳定进入 `Err` 分支；系统不会因为单个任务的 panic 直接崩溃。

## 第二阶段：栈溢出检测与恢复（第 6–9 周）

**目标**：集成 LLVM IR Pass 插桩，实现栈溢出 → unwind 的完整路径 B。

### Week 6–7：LLVM Pass 集成与 __morestack

| # | 任务 | 设计文档章节 |
|---|------|------|
| 2.1 | `.cargo/config.toml` 添加 `-Cllvm-args=-load-pass-plugin=SplitStackPass.so` + `-Cllvm-args=-passes=split-stack` | Section 2.5 |
| 2.2 | 验证编译产物：检查生成的汇编中是否包含 `ldr x9, [x28, #BOUND_TLS_OFFSET]` + `cmp sp, x9` + `b.lo __morestack` | Section 2.2、Section 2.5 |
| 2.3 | 编写 `__morestack` 汇编实现（保存/恢复寄存器 + 修改 LR → `overflow_panic_entry` + ret） | Section 3.2 |
| 2.4 | QEMU 上验证：构造深度递归触发栈溢出，确认 `__morestack` 被调用且寄存器正确恢复 | Section 3.2、Section 2.4 |

### Week 8–9：溢出 Panic 通路

| # | 任务 | 设计文档章节 |
|---|------|------|
| 2.5 | TCB 增加 `TaskUnwindStorage`（预分配 `UnwindException` + `AtomicBool in_use`） | Section 2.4、Section 3.3 |
| 2.6 | 任务创建时初始化 TLS 变量 `__hopter_stklet_bound`（= 栈底 + kGuardMargin），阶段一使用 2KB guard | Section 2.4 |
| 2.7 | 实现 `overflow_panic_entry` 裸函数：读 TLS → TCB → 预分配 exc → 初始化 → 尾调 `_Unwind_RaiseException` | Section 2.4、Section 3.3 |
| 2.8 | QEMU 端到端验证路径 B：栈溢出 → `__morestack` → `overflow_panic_entry` → unwind → `task_run` Err 分支 | Section 2.4、Section 3.2、Section 3.3、Section 4.1 |
| 2.9 | 验证 double panic 检测：在 Drop 实现中故意 panic，确认 `TASK_UNWINDING` 标志阻止递归展开 | Section 5.2 |

**第二阶段里程碑**：栈溢出被提前检测，自动走 unwind 路径完成资源清理，不崩溃。
